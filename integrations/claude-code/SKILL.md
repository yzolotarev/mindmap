# /mindmap (connect-the-dots)

Метод Джастина Санга: знание = схема связей, а не список фактов. Юзер учится, когда
САМ рисует стрелки (Bloom 3-5: применение, анализ, оценка), а не когда читает готовое.
Работает в обе стороны: я объясняю юзеру ИЛИ юзер объясняет мне своё ТЗ.

## Протокол

**Фаза 1 - изоляция (Layer 1, самое крупное).**
1. Если в теме есть `graphify-out/` - сначала прочитать `GRAPH_REPORT.md` (+`graph.json`), узлы брать оттуда.
2. Выделить 4-7 САМЫХ КРУПНЫХ объектов темы, не деталей. Имена 1-3 слова, понятны 15-летнему, жаргон заменить бытовым словом.
3. Очистить экспорт и запустить канвас в фоне:
   ```
   printf '' > /tmp/mindmap-export.txt
   python3 ~/bin/mindmap-canvas.py --nodes "Объект1::Объект2::Объект3" --title "тема"   # run_in_background
   ```
   Когда юзер закроет окно, процесс завершится и меня разбудят; экспорт лежит в `/tmp/mindmap-export.txt`.
4. В чат: та же сцена 2D ASCII (объекты в ┌─┐, разбросаны, БЕЗ линий) + «Объекты изолированы. Соединяй. Канвас открыт.»

**Фаза 2 - проверка (когда пришёл экспорт или юзер описал связи текстом).**
Формат экспорта: `A -> B` стрелка, `A -- B` линия, `=>`/`==` жирная (главная), `: текст` метка связи, `[Группа]: A, B`.

Ответ - ВСЕГДА картинка, никогда не список стрелок текстом («картина стоит тысячи слов» -
явное требование юзера). Нарисовать ASCII-диаграмму боксами `┌─┐`/`│ │`/`└─┘` и стрелками
`──►`/`▲▼◄►`, где исправления - это надписи ПРЯМО НА диаграмме рядом с конкретной стрелкой:
- верная связь юзера - просто оставить как есть, без лишних слов рядом
- пропущенная связь - дорисовать её с пометкой `[+ добавлено: ...]` прямо у стрелки
- ошибочная связь - показать зачёркнутый/неверный вариант юзера и рядом верный с пометкой
  `[✗ было ... / ✓ стало ...]`
Если узел встречается в цикле дважды (агент действует в двух разных фазах) - можно
повторить бокс с тем же именем, это нормальная практика флоучартов, не считается новым узлом.
1. Верные связи - без комментариев на диаграмме (тишина = подтверждение).
2. Ошибки/пропуски - максимум 2 самые важные за раз, оба аннотированы прямо на той же картинке.
3. Один вопрос «почему?» про ключевую стрелку юзера - проверка понимания, а не угадывания.
4. GRINDE-чек схемы: есть группы (G), не иерархия-оглавление (R), есть горизонтальные связи между ветками (I), у стрелок есть направление-логика (D), главное выделено жирным (E).

**Фаза 3 - углубление или фиксация.**
Предложить одно из: раскрыть одну группу в Layer 2 (новый канвас по ней) / сохранить схему заметкой в Obsidian. Не оба сразу.

**Fallback.** «Сдаюсь», «покажи», «не хочу думать» - дорисовать стрелки самому, по одной строке объяснения на стрелку.

## Режим «юзер объясняет мне»

Когда ТЗ неоднозначно: разложить УСЛЫШАННОЕ на объекты, юзер соединяет как он себе представляет,
его схема = спецификация. Расхождения с реальностью кода назвать до начала работы.

## Правила

- Никогда не рисовать готовые стрелки, пока юзер не попробовал сам (кроме fallback).
- Не более 7 объектов на сцене; 4+ ветвей из одной точки = сигнал сгруппировать (chunking).
- Исправлять максимум 2 ошибки за итерацию, иначе перегруз (cognitive load).
- Канвас может быть закрыт без связей - это тоже ответ: спросить, что непонятно в объектах.
